# Multi-turn conversation findings + timing

Generated by driving the **real** Bedrock Converse loop (`evals/conversation.py`)
through two 4-turn sessions against the deterministic fixture data
(`evals/fixture.py`). Model: `us.anthropic.claude-opus-4-8`, region `us-west-2`.

Reproduce:

```bash
python -m evals.conversation                 # generate-from-scratch, then edit
python -m evals.conversation --seed-upload    # start from a hypothetical upload, then edit
python -m evals.conversation --json trace.json # full structured trace
```

## What the agent can do (verified end-to-end, live)

Across both sessions the assistant correctly, with state carried turn-to-turn:

- **Summarized an uploaded constraint file** rule-by-rule (`describe_constraints`),
  including flagging `ABCD` as a non-standard SCAC and a min-of-1 as a weak floor.
- **Drafted a new cap** ("RKNE ≤ 50 for NYC, high priority") — and *previewed the
  scope first* (`preview_constraint_scope` → 8 NYC containers), then warned the cap
  of 50 won't bind because NYC only holds 8 containers. That's real, useful reasoning,
  not just transcription.
- **Edited across turns**: lowered a cap 100→50, bumped priority 80→95, added a
  percent rule, then **deleted** a rule by description ("drop the ABCD minimum-count
  rule") — resolving the natural-language reference to the right index via
  `describe_constraints` then `edit_constraints`.
- **Never over-claimed**: every turn ended with "review and click Apply/Download,"
  never "I applied it." The read-only / never-dispatch contract held.
- **Did light arithmetic reasoning**: "30% of 5 = 1.5 → ~1–2 containers depending
  on rounding," and noticed the new percent rule made the old min-1 rule redundant.

Final working set in both runs: 2 valid constraints, 0 validation problems.

## Timing — the headline

| Session | Turns | Total | Avg/turn | Model round-trips | Tool time |
|---|---|---|---|---|---|
| generate-then-edit | 4 | 46.2s | 11.5s | ~10 | **0.01s** |
| upload-then-edit   | 4 | 39.2s | 9.8s  | 9   | **0.00s** |

**~100% of latency is model round-trips. The Python tool handlers are
negligible — 0–4 ms each.** A turn costs roughly:

```
turn_seconds ≈ (sequential_tool_calls + 1) × ~4.4s per round-trip
```

- 1-tool turn (e.g. just `edit_constraints`) = **2 round-trips** (~6–12s).
- 2-sequential-tool turn (`preview_constraint_scope` → `generate_constraints`) =
  **3 round-trips** (~13s). The preview result has to come *back* to the model
  before it can draft, so it's a full extra ~4.4s.

## Implication for "reusable scripts/tools the agent can leverage"

Because tool execution is free and model round-trips are the entire cost, the
win is **NOT faster tool code or caching tool output** — it's **reducing the
number of sequential model round-trips per turn.** Two concrete levers:

1. **Composite tools that fuse a common 2-call sequence into 1.** The recurring
   pattern is *preview-then-draft* and *describe-then-edit*. A single
   `draft_constraint_with_preview` tool that (a) computes the scope match count
   and (b) validates+stages the rule in one handler would turn the most common
   constraint turn from **3 round-trips → 2** (~30% faster on that turn). The
   model gets the match count back *in the same result* it gets the staged rule,
   so it can still warn "won't bind" without a separate call.

2. **Return richer results so the model doesn't need a follow-up call.**
   `generate_constraints` / `edit_constraints` could include the current scope
   match count for each affected rule in their return payload — then the model
   rarely needs a standalone `preview_constraint_scope` or `describe_constraints`
   round-trip to reason about impact.

Both are low-risk: the underlying pure functions already exist and are tested;
this is composition, not new logic. The harness here (`conversation.py`) is the
measurement tool — re-run it before/after to confirm round-trips/turn drops.

Out of scope for now (bigger changes, only if needed): streaming first-token to
cut *perceived* latency (already wired in `bedrock_client.stream_conversation`);
a smaller/faster model for routing-only turns.

## Optimization shipped + measured (composite constraint tools)

Implemented both levers by enriching the existing tools (no redundant new tool):
`generate_constraints` and `edit_constraints` now accept the loaded data
(`df=`) and fold in a `working_set` — the full indexed staged set with each
rule's `scope_containers` match count. That is exactly what a separate
`preview_constraint_scope` + `describe_constraints` round-trip would have
returned. Tool descriptions were updated to tell the model the counts are
already in the result, so it stops making the follow-up call. Backward
compatible: omit `df` and there's no `working_set` (every existing test/caller
is unaffected). Covered by 6 new tests in `tests/test_chatbot.py`.

**Measured, live, generate-from-scratch scenario (same 4 turns, before → after):**

| Turn | Before | After |
|---|---|---|
| draft 1 rule  | 3 round-trips (preview→generate) | 3 (model still previews a brand-new rule first — its choice) |
| add 2 rules   | 3 (preview×2→generate) | **2 (generate only)** ✓ |
| edit + delete | 3 (describe→edit)      | **2 (edit only)** ✓ |

Two of the three constraint turns dropped from 3 → 2 round-trips — **~one ~5s
round-trip saved per turn (~30% faster on those turns)** — while the answers got
*better*: the model now cites the embedded scope counts inline ("the NYC scope
only matches 8 containers, so a max of 50 won't bind"). When the user explicitly
asks to "check the count first," the model still previews — correct obedience,
not a regression.

Re-measure any time with:

```bash
python -m evals.conversation --json before.json          # generate-from-scratch
python -m evals.conversation --seed-upload --json b2.json # upload-then-edit
# (look at per-turn "model round-trip(s)" in the transcript / model_calls in JSON)
```

## The reusable harness

`evals/conversation.py` is importable and scriptable — not just a CLI:

```python
from evals.conversation import Conversation, hypothetical_upload_df
convo = Conversation()                                  # fixture data by default
convo.seed_uploaded_constraints(hypothetical_upload_df())
t = convo.send("Cap RKNE at 50 for NYC, priority 95.")
print(t.answer, t.tools_used, t.seconds, t.model_calls)
convo.send("Make it 40 and add a 90% floor for ABCD on BAL.")
print(convo.staged_constraints())                       # current working set
convo.print_timing_report()                             # per-tool + round-trip rollup
```

It persists both the Converse message history **and** the streamlit-style
session state across turns, so staged constraints behave exactly as in the
sidebar. Pass your own `working_data` / `rate_data` DataFrames to drive it
against real (or other hypothetical) uploads.
